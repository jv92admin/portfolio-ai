## What Alfred Is

A domain-agnostic LLM orchestration engine published on PyPI as `alfredagain`. You give it a `DomainConfig` (a Python class implementing 23 abstract methods + 50 overridable defaults) and it runs a 5-node LangGraph pipeline that can read, plan, execute, and respond against any database-backed system.

The core insight: the LLM never sees UUIDs. A `SessionIdRegistry` translates between human-readable refs (`recipe_1`, `deal_3`, `gen_recipe_1`) and database IDs. This makes entity tracking deterministic across multi-turn conversations.

## How It Started

Built as a kitchen assistant — recipes, inventory, meal planning, grocery lists, cooking log. 10 entities, 7 subdomains. The orchestration (routing, planning, execution, identity, context) turned out to be domain-agnostic. Extracted into `alfred-core`, published on PyPI.

## How It Works

### The Graph (5 Agents)

Each agent is an async function that receives `AlfredState` (50+ fields) and returns state updates.

| Agent | Job | Key Decision |
|-------|-----|-------------|
| **Understand** | Resolve entity references ("that recipe" -> `recipe_1`), curate what stays active, detect quick-mode | Can short-circuit to Reply if query is simple read |
| **Think** | Decompose goal into steps (read/analyze/generate/write), assign parallel groups | `plan_direct` (execute), `propose` (ask first), `clarify` (need info) |
| **Act** | Execute steps via CRUD tools + custom tools, register entities, track batch progress | Loops per step, executes in parallel within groups |
| **Reply** | Synthesize response, format for humans, show batch status | Tags generated content as [unsaved] |
| **Summarize** | Record what happened (audit ledger), compress older turns, persist registry | No LLM call — deterministic bookkeeping |

**Routing flow:**
- Simple reads skip Think entirely (Understand detects quick-mode -> Act -> Reply)
- Complex tasks go full pipeline (Understand -> Think -> Act loop -> Reply -> Summarize)

### Entity Identity (SessionIdRegistry)

Single source of truth for entity tracking across turns. 1,166 lines.

- **Refs:** `recipe_1`, `inv_5`, `meal_plan_2` (database entities), `gen_recipe_1` (LLM-generated, unsaved)
- **Counters:** Monotonically increasing per type, never reset in session
- **Bidirectional:** `ref_to_uuid` and `uuid_to_ref` maps
- **Metadata:** action tags (read/created/generated/linked), labels, types, turn tracking, detail level
- **Curation:** Understand decides what older entities stay active with retention reasons

**4 registration paths:**
1. Read from DB -> `translate_read_output()` -> assigns ref
2. Generated by LLM -> `register_generated()` -> `gen_*` ref, `__pending__` UUID
3. Saved to DB -> `register_created()` -> promotes `gen_*` to real UUID
4. User action in UI -> `register_from_ui()` -> tagged `:user`

### The Approval Model

Generated content lives in memory with `gen_*` prefix. Nothing writes to the database without explicit user promotion. Flow:

1. LLM generates content -> `gen_recipe_1` with `__pending__` UUID
2. Reply shows it tagged [unsaved]
3. User says "save it" (or "save with changes")
4. Act creates DB record, promotes `gen_*` to real UUID
5. Summarize clears promoted artifacts at turn end

### Context Layers (3 Independent Layers)

| Layer | What | Built By | Used By |
|-------|------|----------|---------|
| **Entity** | Refs, labels, lifecycle state, detail tracking | SessionIdRegistry (deterministic) | All nodes |
| **Conversation** | Last 3 turns full text, older turns compressed, engagement summary | Summarize node | Think, Reply |
| **Reasoning** | What was planned, executed, changed — audit trace | Summarize node (deterministic) | Think (last 2 turns), Reply (last turn) |

Compression thresholds: 3 full turns, 3 full step results, 8K tokens for Think context, 25K for Act.

### How a Domain Plugs In (The Injection Map)

A domain implements `DomainConfig` and provides:

| What | Purpose | Kitchen Example | CRM Example |
|------|---------|----------------|-------------|
| **Entities** | What exists in the DB | 10 types (recipes, inventory, meals...) | 13 types (deals, contacts, leads, estimates...) |
| **Subdomains** | How tables group by user intent | 7 (inventory, recipes, shopping, meal_plans, tasks, preferences, history) | 6 (crm, estimating, design, construction, billing, plants) |
| **Personas** | How the LLM thinks per subdomain | Recipe assistant, meal planner | Sales coach, estimator thinking in margins, PM thinking in schedule |
| **Schemas** | Table structures, field enums | Cuisine types, difficulty levels | Deal statuses, lead sources |
| **Middleware** | Pre/post hooks on CRUD | Semantic search, auto-include nested relations, ingredient lookup | Cents->dollars conversion, blocked fields, audit trail |
| **Custom tools** | Domain-specific actions | ingredient_lookup, semantic_search | send_email |
| **Bypass modes** | Skip the graph for special flows | cook mode, brainstorm mode | (none) |

**Registration is one line:**
```python
register_domain(KITCHEN_DOMAIN)  # runs at import time
```

### What Makes It Different

1. **Entity identity is deterministic** — refs assigned by CRUD results, not LLM guesses. Counters never reset. `recipe_1` always means the same thing.
2. **Approval model** — `gen_*` prefix tracks unsaved content. Nothing writes without user say-so.
3. **Context is layered, not dumped** — Each node gets only what it needs. Think gets entity + conversation + reasoning. Act gets entity + step context. Reply gets everything.
4. **Middleware intercepts CRUD** — Domains hook into read/write lifecycle. Kitchen does semantic search and ingredient lookup. CRM does cents-to-dollars and field blocking.
5. **Table aliasing** — CRM has 160+ aliases so the LLM can say "project" instead of "construction_projects" and it still works. Prevents LLM errors at the adapter level.
6. **Subdomain personas** — Same LLM, different expertise per context. A billing query gets "office manager paranoid about cash flow." An estimating query gets "experienced estimator thinking in margins."

## Landscaping CRM Integration (Key Details)

The CRM domain proves Alfred works at enterprise scale:
- 13 entity types across 40+ tables
- 6 subdomains with specialized personas
- JWT-scoped Supabase adapter with RLS for multi-tenancy
- 160+ table/field aliases for LLM error tolerance
- Pre-write middleware strips blocked fields (QuickBooks sync, computed status), sets audit trails
- Cloud function side effects on status changes (deal -> auto-creates design card)
- Custom `send_email` tool

## Kitchen Domain (Reference Implementation)

The original domain that proved the architecture:
- 10 entities, 7 subdomains
- Semantic search via pgvector embeddings
- 4-tier ingredient lookup (exact -> aliases -> fuzzy trigram -> semantic)
- Auto-include nested relations (recipe -> recipe_ingredients)
- Bypass modes: cook mode (frozen recipe context, 1 LLM call/turn) and brainstorm mode (read-only creative iteration)
- Onboarding flow: dietary restrictions -> household profile -> cooking vibes
- Frontend: React + Supabase Auth + SSE streaming
